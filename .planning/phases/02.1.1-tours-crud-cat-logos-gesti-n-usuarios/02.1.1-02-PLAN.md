    1. Create `apps/tours/web/src/app/api/_lib/proxy.ts` (new) — single shared helper for Route Handlers. The full implementation is in `02.1.1-PATTERNS.md` §8 and is also documented inline below as directive prose (no fenced code per planner rules).

       Behavior: export one async function `proxyJson(path: string, method: string, body?: string): Promise<Response>`. Internally: (a) call `getServerSession(authOptions)` and extract `token = (session as any)?.token?.token`; (b) build the upstream URL as `${API_URL}${path}` where `API_URL = process.env.TOURS_API_URL || "http://tours-api:8000"` (same default the existing route handlers use); (c) call `fetch(API_URL + path, { method, headers: { "Content-Type": "application/json", ...(token ? { Authorization: "Bearer " + token } : {}) }, body, cache: "no-store" })`; (d) return a `new Response(await res.text(), { status: res.status, headers: { "Content-Type": "application/json" } })` so the status + body pass through verbatim (the FastAPI `detail` field stays a JSON object, not a stringified blob).

       The `body` parameter is `string | undefined` (already-serialized text from `req.text()`). For GET with no body, omit the param. For POST/PUT/DELETE with no body (e.g. the restore endpoint), pass `undefined` — `fetch` handles that fine.

       Why a shared helper: 5 new Route Handlers all need the same fetch-with-bearer pattern; without it, 5 copies of the same 10 lines diverge over time. The 3 existing handlers (ventas POST, simular GET, catalogos GET) become 3-5 line wrappers after refactor.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Next.js Route Handler | Same-origin. Browser sends session cookie; Route Handler reads session via getServerSession, extracts JWT, forwards as `Authorization: Bearer` to FastAPI. No cross-origin CORS surface (D-02). |
| Next.js Route Handler → FastAPI | Docker internal DNS (`tours-api:8000` on `traefik-public` network). FastAPI is not exposed to internet (no Traefik router labels — D-01). JWT verified in FastAPI via `pyjwt.decode` with shared `NEXTAUTH_SECRET` — protects against forged tokens. |
| User → Modal form | All client-side validation is UX. Backend re-validates (Pydantic min_length + extra="forbid" on UsuarioUpdateIn). |
| Admin → DELETE /usuarios | Protected by `require_role("admin")` in the router + D-11 last-admin check + D-12 self-delete check + D-10 email unique check. |

The 8 in-scope items from CONTEXT are all mitigated. The threat register below covers the changes in Plan 02.

The PasswordForm submits to `/api/usuarios/me/password` which is itself guarded by `get_current_user` (any authed). A user can only change THEIR OWN password — the route uses `current_user["id"]` from the JWT, not from the body. There is no admin override here. |

A 401 is returned for wrong current. The route does NOT call bcrypt.checkpw on the JWT bearer — it uses the session-derived `current_user["id"]` to fetch the user from DB and compares against that user's password_hash. |
