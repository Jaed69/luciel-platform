# Phase 02.1.1: Tours — CRUD catálogos + gestión usuarios - Context

**Gathered:** 2026-07-04
**Status:** Ready for planning

<domain>
## Phase Boundary

Completar el CRUD funcional de catálogos (backend solo tiene GET + DELETE, frontend tiene botones sin acción) y construir gestión de usuarios desde cero (no existe endpoint `/usuarios` en backend, página actual usa datos hardcodeados). Ajusta el modelo de permisos de Phase 02.1: `contabilidad` ahora puede editar catálogos (no solo admin). Conserva el patrón existente: soft-delete, audit log automático, RBAC, JWT bridge, internal FastAPI.

**In scope:**
- Backend `PUT /catalogos/{entidad}/{id}` para editar codigo/nombre
- Backend `/usuarios` CRUD completo (GET list + POST create + PUT edit + DELETE soft) — todos los POST/PUT hashean password con bcrypt
- Backend `PUT /usuarios/me/password` para self-service password change (cualquier usuario autenticado)
- Backend check de integridad referencial antes de DELETE catalogos: si hay registros activos referenciando el id → 409 Conflict con `{"detail": "...", "referencias": [...]}`
- Ajuste `require_role` para CRUD catalogos: ahora acepta `["admin", "contabilidad"]` (no solo admin) — D-24 modificado
- Frontend `<CatalogoFormModal>` genérico (campos codigo + nombre) reusado por las 5 entidades: agencias, tours, vendedores, formas-pago, monedas
- Frontend `<UsuarioFormModal>` para crear/editar usuarios (admin only)
- Frontend `/perfil` página para cambio de contraseña self-service
- Frontend `admin/usuarios` page conectada a API real (reemplaza mockup)
- Frontend action "Restaurar" para reactivar items con `activo=false` (soft-delete reversal)
- Route Handlers Next.js: POST/PUT/DELETE para catalogos, todos los verbs para usuarios
- Ajuste del nav: contabilidad ve link Catálogos (no solo admin)

**Out of scope (deferred):**
- Email notifications al crear usuario
- `must_change_password` flag (password es válido desde creación)
- Password reset por email (solo admin via PUT /usuarios/{id})
- Multi-campo en catálogos (agencias con RUC, tours con duración, etc.) — YAGNI, JSONB metadata si surge
- Bulk import / export de catálogos
- Avatar / foto de perfil
- 2FA / MFA
- Audit log de la propia gestión de usuarios (ya cubierto por el before_flush listener — el flujo es automático)

<decisions>
## Implementation Decisions

### Formularios de catálogos

- **D-01:** **Formulario genérico** con dos campos (`codigo` + `nombre`) para las 5 entidades (agencias, tours, vendedores, formas-pago, monedas). Un solo componente `<CatalogoFormModal>` reutilizado por agregar y editar — diferencia solo el handler submit (POST vs PUT) y los datos pre-cargados. El schema `CatalogoIn` ya existe en `app/schemas/core.py` y tiene exactamente esos dos campos. Sin campos específicos por entidad en esta fase.
- **D-02:** **PUT unificado `PUT /catalogos/{entidad}/{id}`** sigue el mismo patrón que `PUT /cuentas/{cuenta_id}` (line 64 de `app/routers/core.py`) — recibe `CatalogoIn`, actualiza codigo/nombre, mantiene `activo` intacto. RBAC: `require_role(["admin", "contabilidad"])` (cambio vs D-24 de Phase 02.1).
- **D-03:** **POST y PUT nunca tocan `activo`** — solo codigo/nombre. Para activar/desactivar se usa el endpoint DELETE existente (soft-delete) o un futuro POST /catalogos/{entidad}/{id}/restore (si el UI lo pide; sino el mismo PUT no permite cambiar `activo`).

### Gestión de usuarios

- **D-04:** **Endpoint `POST /usuarios` recibe password en texto plano en el body, backend hashea con bcrypt antes de persistir.** Schema `UsuarioCreateIn`: `{email, username, password, rol}`. No email notification. No flag `must_change_password`. El usuario entra con esa password y la cambia cuando quiera desde su perfil.
- **D-05:** **Endpoint `GET /usuarios` devuelve lista con `id, email, username, rol, activo, creado_en, ultimo_acceso`** (excluye `password_hash` siempre — D-26 redaction). Paginación NO requerida en esta fase — volumen bajo (decenas de usuarios, no miles).
- **D-06:** **Endpoint `PUT /usuarios/{id}` para editar email, username, rol, activo** — NO permite cambiar password (eso va por endpoint dedicado `PUT /usuarios/me/password` o admin resetea con endpoint separado si lo necesita). RBAC: admin only.
- **D-07:** **Endpoint `DELETE /usuarios/{id}` soft-delete (activo=false).** RBAC: admin only. No hard-delete. La auditoría queda intacta.
- **D-08:** **Self-service password change: `PUT /usuarios/me/password` con body `{current_password, new_password}`.** Backend verifica current_password contra bcrypt hash, hashea new_password, actualiza. Cualquier usuario autenticado puede usarlo. NO RBAC check más allá de autenticación. Audit log captura el cambio via before_flush (D-21) con `password_hash` redactado (D-26).
- **D-09:** **Admin password reset: `PUT /usuarios/{id}/password` con body `{new_password}`.** Backend hashea, actualiza. Admin only. NO requiere current_password (es reset, no cambio). Distinto de D-08.
- **D-10:** **Validación de email único en backend** — la columna `usuarios.email` ya tiene UNIQUE constraint (model `core.py:74`). FastAPI captura IntegrityError → 409 Conflict.
- **D-11:** **Protección de "último admin"** — antes de DELETE /usuarios/{id} o PUT /usuarios/{id} (cambio de rol away from admin), backend verifica que queda al menos 1 admin activo. Si no → 409 Conflict. Previene lockout. Mismo patrón que ya está en el frontend (line 42 de `admin/usuarios/page.tsx`).
- **D-12:** **Protección de auto-eliminación** — backend rechaza DELETE /usuarios/{id} si `id == current_user.id` → 409 Conflict con mensaje "no puede eliminar su propia cuenta". Frontend ya muestra el cursor-not-allowed pero el backend es la fuente de verdad (D-26 structural).

### Permisos por rol (revisión 02.1.1)

- **D-13:** **CRUD catálogos** ahora acepta `admin` O `contabilidad` (cambio vs D-24 de Phase 02.1 que restringía a admin). Esto desbloquea a contabilidad para mantener catálogos sin depender de admin. `vendedor` sigue sin acceso.
- **D-14:** **Audit log sigue admin-only** (D-24 se mantiene en su totalidad — contabilidad NO ve audit_log).
- **D-15:** **CRUD usuarios sigue admin-only** (sin cambio vs Phase 02.1).
- **D-16:** **Dashboard saldos/tours pendientes** — contabilidad sigue con acceso read-only (Phase 02.1 ya lo tenía). Vendedor sigue forzado a `vendedor_id = user.id` (T-02.1-14 ya implementado).
- **D-17:** **Liquidaciones** siguen admin-only y contabilidad puede ver (read). Vendedor NO accede. Sin cambio.

### Integridad referencial en catálogos

- **D-18:** **Bloqueo duro (409 Conflict) al desactivar (soft-delete) un item de catálogo que está siendo referenciado por registros activos.** Antes de `DELETE /catalogos/{entidad}/{id}`, el backend hace un query de referencias en las tablas que tienen FK a la entidad correspondiente. Si `count > 0` en alguna de esas tablas, rechaza con 409. Body: `{"detail": "No se puede desactivar — N registros lo referencian", "referencias": [{"tabla": "tours_servicios", "count": 12}, ...]}`.
- **D-19:** **Tablas que cuentan como referencia activa para cada entidad:**
  - `agencias` → `tours_servicios.agencia_id WHERE activo=1` (o `liquidado IS NULL`? — planner decide: si la venta existe, la agencia existe; no se puede borrar lo que ya se vendió)
  - `vendedores` → `tours_servicios.vendedor_id WHERE activo=1`
  - `tours` → `tours_servicios.tour_id WHERE activo=1`
  - `formas-pago` → `tours_servicios.forma_pago_id WHERE activo=1`
  - `monedas` → `tours_servicios.moneda_id WHERE activo=1` + `cuentas.moneda` (no se puede borrar PEN/USD si hay cuentas en esa moneda)
  - Planner refina — el principio es: si hay historia contable, no se borra.
- **D-20:** **El bloqueo aplica también a PUT que cambia `activo` si llegara a permitirse** (en esta fase PUT NO cambia activo — D-03). El check solo se ejecuta en DELETE por ahora.
- **D-21:** **Front muestra la lista de referencias en la respuesta 409** — el toast/alert del componente DELETE muestra el `detail` + lista de tablas/recuentos para que el admin sepa qué tours/vendedores/agencias se verían afectados. No es automático, es informativo.

### the agent's Discretion
- Schema `UsuarioCreateIn` / `UsuarioUpdateIn` / `UsuarioOut` exact shape (campos, validators) — planner define siguiendo el patrón de `CatalogoIn` / `CatalogoOut`.
- Validaciones específicas (email regex, password min length, username min length) — planner elige defaults razonables (email regex estándar, password min 8 chars, username min 3 chars).
- Componente exacto para password change en `/perfil` (puede reusar `FloatingLabelInput` + `Button` existente, sin modal).
- Estructura del toast/error en el frontend al recibir 409 con referencias — el componente `Toast.tsx` existe, planner decide si necesita variant o sub-componente para listas.
- Si la acción "Restaurar" merece su propio endpoint (`POST /catalogos/{entidad}/{id}/restore`) o si el mismo PUT debería poder cambiar `activo` — planner decide. YAGNI: un endpoint dedicado es más simple que sobrecargar PUT.
- Cómo se pasa `password` en `POST /usuarios` y `PUT /usuarios/{id}/password` — planner elige si HTTPS-only (ya garantizado por Traefik LE) o agregar constraint adicional.
- Auth flow para self-service password change: ¿el endpoint rechaza si el token JWT es de un usuario que no existe más? (cubierto por `get_current_user` dependency que falla 401 si no encuentra el usuario — no requiere trabajo extra).

<specifics>
## Specific Ideas

- "Sandra" del enunciado original del mini-ERP — la admin que opera el sistema. Al pedir "self-service para vendedor", se entiende que cualquier vendedor puede ir a `/perfil` y cambiar su propia contraseña. No es un "vendedor cualquiera" — es cualquier usuario autenticado. La acción está disponible para los 3 roles.
- "Contabilidad puede editar catálogos" — viene de la observación operacional de que contabilidad mantiene las tablas de cuentas/agencias/etc. y no debería depender de admin para alta/baja de items. Esto es un refinamiento del modelo de Phase 02.1, no un cambio radical.
- "No bloquear al admin" — D-11 (último admin) y D-12 (auto-eliminación) son protecciones pero NO bloquean al admin de gestionar otros admins. Solo bloquean el caso patológico (quedarse sin admins, o eliminarse a sí mismo).
- "Restaurar" — el soft-delete puede revertirse. La página actual muestra "Restaurar" link solo cuando `activo=false` (line 67-69 de `catalogos/[entidad]/page.tsx`). Phase 02.1.1 conecta este link a una acción real. Planner decide si es `PUT /catalogos/{entidad}/{id}` con `activo=true` (más simple) o endpoint dedicado `POST /catalogos/{entidad}/{id}/restore` (más explícito).
- "Toast con referencias" — el 409 Conflict con `referencias: [{tabla, count}]` puede renderizarse en el toast. La lista es informativa, no accionable (el admin no puede forzar — debe ver los tours afectados y decidir si reabre liquidaciones, etc.).
- "Validación client-side ANTES del submit" — los modales deben validar antes de mandar al backend (ej: email formato, password min length) para evitar round-trips innecesarios. Backend sigue siendo la fuente de verdad (re-validates).
- "Email único" — el backend rechaza duplicados con 409. El frontend puede mostrar el error en el form (campo email con mensaje de error) sin abrir un toast genérico.
- "Confirmación antes de DELETE" — `window.confirm` simple o modal de confirmación. El componente `Modal` puede usarse. Para "Restaurar" no requiere confirmación (operación reversible).
</specifics>
