    Create `.github/workflows/release.yml` — the CI/CD pipeline that builds the arm64 landing image and pushes to GHCR.

    **Workflow trigger:**
    - on.push.branches: [main]
    - on.push.paths: ['apps/landing/**', '.github/workflows/release.yml']

    **Environment variables:**
    - REGISTRY: ghcr.io
    - IMAGE_NAME: ${{ github.repository_owner }}/luciel-platform-landing

    **Job: build-and-push**
    - runs-on: ubuntu-latest
    - permissions: contents read, packages write
    - Steps:
      1. actions/checkout@v4
      2. docker/setup-qemu-action@v3 (arm64 emulation — per D-06, GitHub runners are x86 only)
      3. docker/setup-buildx-action@v3
      4. docker/login-action@v3 with registry ghcr.io, username ${{ github.actor }}, password ${{ secrets.GITHUB_TOKEN }}
      5. docker/metadata-action@v5 — tags: type=sha,prefix=sha- and type=raw,value=latest,enable={{is_default_branch}}
      6. docker/build-push-action@v6 — context must be repo ROOT (not ./apps/landing) because the Dockerfile uses multi-stage with pnpm workspace (COPY paths reference root pnpm-lock.yaml). Set context to '.' and file to './apps/landing/Dockerfile'. platforms: linux/arm64. push: true. tags and labels from metadata step. cache-from: type=gha, cache-to: type=gha,mode=max.

    CRITICAL: The Dockerfile in Plan 01-01 Task 1 is structured for a root-level build context (it COPYs root package.json + pnpm-lock.yaml for pnpm workspace). The workflow build context must be '.' (repo root) with the Dockerfile path specified as './apps/landing/Dockerfile'. If the executor finds the Dockerfile uses a local context (apps/landing only), adjust context to './apps/landing' instead — verify by reading the Dockerfile first.

    **Image naming must match docker-compose.yml:** The IMAGE_NAME env var must produce the same image reference as `ghcr.io/${GITHUB_USER}/luciel-platform-landing:latest` in docker-compose.yml. Since github.repository_owner equals the GitHub username/org, this aligns automatically.
  </action>
