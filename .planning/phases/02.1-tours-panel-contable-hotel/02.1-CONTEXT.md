## Phase Boundary

Sistema contable multi-usuario (mini-ERP modular) para hotel/agencia de tours de Cusco con **core de partida doble** + **módulo Tours** (catálogo + liquidaciones + comisiones), auth por roles, audit log estructural y dashboard mínimo. Reemplaza el Excel con macros VBA que borra sin rastro. Accessible en `https://tours.luciel.dev` con cert Let's Encrypt propio. Primer inquilino de un mini-ERP modular — el módulo café/Basílica se agrega después sin rehacer el core.

**In scope (TOURS-01..10):**
- `apps/tours/` con Next.js 16 + TypeScript App Router + Tailwind (frontend) y FastAPI + SQLAlchemy async + SQLite WAL (backend)
- Traefik router para `tours.luciel.dev` con cert LE propio (HTTP-01, per-subdomain, no wildcard cert)
- Auth NextAuth/Credentials + bcrypt + tabla `usuarios` con roles `admin` / `vendedor` / `contabilidad`
- Core contable de partida doble: `cuentas` / `asientos` / `asiento_lineas` con balance validado en FastAPI dentro de la transacción (no SQL trigger)
- Módulo Tours: `tours_catalogo` / `liquidaciones` / `tours_servicios` (con `asiento_id` FK al core, `metadata` JSONB híbrido) — separado del core
- `comision_reglas` con prioridad determinística 1-4 + endpoint `/simular`
- `audit_log` con `datos_antes` / `datos_despues` por evento INSERT/UPDATE/DELETE
- Liquidaciones: abrir + cerrar; el cierre genera asientos de distribución/comisión automáticamente
- Dashboard mínimo: filtros por fecha/agencia/vendedor/moneda + tabla + totales de saldo por cuenta (sin gráficos)
- Catálogos CRUD via admin UI (agencias, tours, vendedores, formas de pago, monedas) — no texto libre
- 2 plans: 02.1-01 scaffold+CRUD+posting al ledger ; 02.1-02 liquidaciones+dashboard+deploy

**Out of scope (deferred a 2.1.x o posteriores):**
- Full dashboard con charts + export Excel/PDF (2.1.x)
- PWA (YAGNI — deferred)
- Migración de datos Excel (arranque limpio)
- `packages/ui` compartido a nivel workspace (local en `apps/tours/`, YAGNI hasta 2+ apps)
- Module registry framework (YAGNI — módulos nuevos se agregan por convención de carpetas, no metadata)
- Módulo café/Basílica (fase posterior — no rehacer el core)
- Conciliación bancaria automatizada
- Reportes contables completos (balance general, estado de resultados) — dashboard mínimo en 2.1

## Implementation Decisions

### Topología de deploy

- **D-01:** **2 contenedores compose** — `tours-web` (Next.js 16 standalone) y `tours-api` (FastAPI + uvicorn) como servicios separados. Traefik enruta `Host=tours.luciel.dev` sólo al frontend. FastAPI queda interno en `traefik-public` (sin router Traefik expuesto a internet); el frontend lo alcanza vía DNS interno `http://tours-api:8000`. SC#3 satisfecho con router único en el compose.
- **D-02:** **Auth bridge via Server Components fetch interno + JWT verificado en FastAPI.** Next.js Server Components / Route Handlers hacen fetch a `http://tours-api:8000` leyendo el session token (JWT firmado con `NEXTAUTH_SECRET`) y lo envían como `Authorization: Bearer <jwt>`. FastAPI valida el mismo `NEXTAUTH_SECRET` con `pyjwt` (no requiere portar NextAuth). Stateless, sin cookie cross-domain, sin CORS. El token nunca se expone al browser. Evita exponer FastAPI a internet.
- **D-03:** **Volume dedicado `tours-db-data`** (named volume en compose). SQLite WAL vive en `/data/tours.db` dentro del contenedor `tours-api`. Cada app con su propio volume + su propio lifecycle/backup. Módulo café futuro = su propio volume (alineado con PROJECT.md "apps aisladas"). Evita acoplamiento entre módulos del mini-ERP.
- **D-04:** **CI: extender `release.yml` existente como matrix por app.** Convertir el workflow actual (mono `landing`) en `matrix: app: [landing, tours-web, tours-api]` con build context + image name variables desde pnpm-workspace packages. 2-3 jobs paralelos → imágenes GHCR: `ghcr.io/${GITHUB_USER}/luciel-platform-tours-web` y `luciel-platform-tours-api`. Escala limpio a `graph.luciel.dev`, `hackathons.luciel.dev` sin tocar el workflow por cada nueva app. Imagen scheme: `:sha-<short>` + `:latest` (mantiene Phase 1).
- **the agent's discretion:** Dockerfile FastAPI (python:3.13-slim + uvicorn CMD + uv install), healthcheck internals, nginx config del front (heredar de landing), secrets/path envs del backend (`NEXTAUTH_SECRET` compartido entre front y api, `DATABASE_URL=/data/tours.db`, `JWT_ALGORITHM=HS256`). El planner fija defaults saneados.

### Multi-moneda (PEN/USD)

- **D-05:** **Una cuenta por moneda en el chart de cuentas.** `101-CAJA-PEN`, `101-CAJA-USD`, `401-INGRESOS-TOURS-PEN`, `401-INGRESOS-TOURS-USD`, `501-COSTOS-TOURS-PEN`, etc. Cada cuenta nace con `moneda CANÓNICA` NOT NULL (columna `cuentas.moneda`). `asiento_lineas.monto` siempre en la moneda de la cuenta — sin columnas extra, sin conversiones implícitas. Cada asiento es **single-moneda**: el balance = 0 se valida dentro de la misma moneda; un asiento no puede mezclar cuentas de monedas distintas. SC#7 satisfecho naturalmente.
- **D-06:** **Conversión USD→PEN via operación interna de TC preferencial on-demand.** No hay tabla `tipo_cambio` ni sincronización con SBS/Sunat. Cuando la empresa necesita soles, la admin registra un asiento del core con `tipo=tc_interno`: débito `101-CAJA-PEN` (monto PEN al TC preferencial elegido), crédito `101-CAJA-USD` (monto USD convertido), y débito/crédito `672-GAN/PERD-TC` por la diferencia. El TC elegido + justificación van en `asientos.metadata` (JSONB híbrido). Es会计师operacional, no automatizado. Los Tours se vende y cobra en su moneda original; la conversión es un evento contable discreto.
- **D-07:** **Comisión se calcula y distribuye en PEN (soles) únicamente.** Margen bruto del tour = `ingreso−costo`. Los ingresos en USD y costos en PEN se acumulan en sus cuentas originales hasta que una operación de TC interna los convierte a PEN. La comisión final se liquida en soles sobre el margen convertido.
- **D-08:** **`asiento_lineas` requiere `cuentas_id` con FK válida + `cuentas.moneda` determina la moneda de la línea implícitamente.** No hay columna `moneda` en `asiento_lineas` (se infiere de la cuenta). Esto evita inconsistencias (línea con monto USD apuntando a cuenta PEN). Validación en FastAPI dentro de la tx junto al balance.

### Modelo de comisiones y liquidaciones

- **D-09:** **Comisión = ganancia/margen del tour** (ingreso − costo proveedores), no porcentaje sobre venta bruta.
- **D-10:** **Reparto empresa/vendedor configurable por vendedor.** Default 50/50 pero variable según acuerdo (perfil del vendedor / acuerdo con admin). Se modela con `comision_reglas` y precedencia 1-4: `vendedor+tour > vendedor > tour > default global`. La regla default global (50/50) es **no borrable**.
- **D-11:** **Liquidación = cierre de caja de un rango de fechas** (desde/hasta), opcionalmente filtrable por agencia/vendedor. Estado: `abierta → cerrada → reverted` (3 estados). La admin abre, el sistema consolida `tours_servicios` del rango con `liquidacion_id IS NULL`, calcula margen por vendedor, resuelve `comision_reglas` por precedencia, y genera asientos al cierre.
- **D-12:** **Comisiones a cuenta por pagar (201-COMISIONES-POR-PAGAR), no afecta caja en el cierre.** El cierre genera: débito `501-COSTOS-COMISIONES` (gasto), crédito `201-COMISIONES-POR-PAGAR` (obligación). La admin luego hace un pago manual separado: débito `201` / crédito `101-CAJA-PEN`. Separa el evento contable del flujo de caja real — conciliación limpia, el pago no afear el cierre.
- **D-13:** **Reopen con asientos de reversión + recálculo.** Si se agregan tours al rango después del cierre, la admin reabre la liquidación: el sistema genera asientos de reversión que cancelan los del cierre original (la liquidación original pasa a estado `reverted`, visible en audit_log), desbloquea los tours del rango, permite edición, y al volver a cerrar genera nuevos asientos en una nueva instancia de liquidación. Ortodoxo contablemente, trazable, no destructivo. La liquidación original queda revertida y visible; no se hace edición inline de asientos cuadrados.
- **D-14:** **Tours bloqueados mientras están assignados a liquidación cerrada.** `tours_servicios` con `liquidacion_id` apuntando a liquidación en estado `cerrada` no se puede editar ni borrar. Para corregir hay que reabrir la liquidación (revierte asientos + desbloquea). Previene asientos inconsistentes (tour editado sin propagar al asiento). El bloqueo se valida en FastAPI dentro del servicio de update/delete de tours.
- **D-15:** **FK `asiento_id` en `liquidaciones` (1 liquidación → N asientos al cierre) y en `tours_servicios` (1 tour → 1 asiento de venta).** La liquidación guarda su(s) asiento(s) de cierre para navegación; el tour guarda su asiento de venta al momento de registrar. Dos FKs distintos, dos instantes distintos.
- **D-16:** **ID único visible por liquidación: `LIQ-AAAA-NNN`** (ej. `LIQ-2026-001`). Generado al cerrar. Columna `codigo` en `liquidaciones`. Referencia en reportes, pagos a vendedores y conciliación.
- **D-17:** **Pre-checks validan antes de cerrar el cierre aborta con lista de problemas.** Validaciones: (a) todos los tours del rango tienen costo cargado — si falta, aborta con lista `T-100, T-102, T-104`; (b) operaciones internas TC necesarias para convertir USD→PEN en el rango están registradas (si no, advierte qué tours no convertirán); (c) cada vendedor en el rango tiene `comision_regla` resoluble (asume default global 50/50 si no existe específica). El cierre no procede si cualquier check falla. Previere liquidaciones erróneas — SC#8 (no más errores del VBA).
- **D-18:** **Un tour, una liquidación.** `tours_servicios.liquidacion_id` NULL hasta entrar en un cierre; luego FK a una única liquidación. Un tour en liquidación cerrada no puede entrar en otra — hay que reabrir la original si necesita ajuste. Consistente con D-14 (bloqueo).
- **D-19:** **Mover tour entre liquidaciones cerradas: no permitido sin reabrir la original.** Reabre original (revierte + desbloquea), quita tour, cierra original sin ese tour, crea nueva liquidación incluyendo el tour. Procedimiento cuidadoso pero trazable, cada paso en audit_log.
- **D-20:** **Dashboard marca tours no liquidados** (con `liquidacion_id IS NULL`) ordenados por fecha con indicador "días desde venta". La admin decide cuándo cerrarlos. No se fuerza el cierre, no hay scheduler. Day-to-day operacional via el dashboard de SC#7.

### Audit log

- **D-21:** **Mecanismo: SQLAlchemy event listener `before_flush` (async).** Captura automáticamente any INSERT/UPDATE/DELETE de tablas con mixin `Auditable`. No depende de que cada dev acuerde acordarse de llamar `audit_log.record()`. Solución estructural al problema del VBA que borra sin rastro — el mecanismo no puede olvidarse. Cumple TOURS-08 estructuralmente.
- **D-22:** **Cobertura: todas las tablas de negocio con mixin `Auditable`.** `usuarios`, `contactos`, `cuentas`, `asientos`, `asiento_lineas`, `modulos`, `tours_catalogo`, `liquidaciones`, `tours_servicios`, `comision_reglas` (y cualquier catálogo CRUD: agencias, vendedores, formas_de_pago, monedas). No se audita `audit_log` a sí misma (auto-referencia). Volumen bajo a esta escala, no es problema.
- **D-23:** **Usuario_id via `ContextVar` + FastAPI dependency injection.** Dependency que setea `ContextVar` al validar el JWT. El event listener SQLAlchemy lee el `ContextVar` al grabar audit_log. Compatible con async SQLAlchemy, no depende de session state (SQLite no tiene), patrón standard en FastAPI. Evita pasaje explicita en cada service method.
- **D-24:** **Permisos: solo role `admin` ve audit_log.** SC#2 específica "vendedor no ve totales globales ni audit log" — se cumple al pie. Contabilidad no lo ve tampoco en 2.1 (auditor natural pero SC#2 es restrictiva). Revisión de permisos de contabilidad en 2.1.x si la operación real lo pide.
- **D-25:** **Schema: `audit_log(id, tabla, registro_id, operacion 'I'/'U'/'D', datos_antes TEXT NULL, datos_despues TEXT NULL, usuario_id, timestamp)`.** Una fila por fila afectada. `datos_antes`/`datos_despues` como TEXT con JSON serializado (SQLite no tiene JSONB nativo, json.dumps es suficiente). Para INSERT: `datos_antes=NULL`, `datos_despues={...row...}`. Para DELETE: viceversa. Para UPDATE: ambos. Index en `(tabla, registro_id, timestamp)` para query por entidad.
- **D-26:** **Campos sensibles redactados en el snapshot.** El mixin `Auditable` define lista de campos sensibles (inicialmente `usuarios.password_hash`) que se nullifican en `datos_antes`/`datos_despues` antes de serializar. La auditoría registra que alguien cambió el password (quién y cuándo) sin exponer el contenido. SC#8 cumple de forma segura — audit_log no es otro vector de leaks.

## Specific Ideas

- "Reemplaza la suma manual de Sandra en la fila Egreso" (ROADMAP Phase 02.1, success criterion #5) — referencia directa al Excel actual. El cierre de liquidación debe generar visualmente la misma output que Sandra esperaría ver, pero automaticamente. El planner puede agregar al reporte del cierre el equivalente "fila Egreso" con breakdown por vendedor para facilitar migración UX.
- Contraseña hardcodeada "2808" del VBA (ROADMAP Phase 02.1, success criterion #2 context) — reemplazada por NextAuth/Credentials + bcrypt. La validación con bcrypt + tabla `usuarios` permite auditoría por usuario (D-21 audit_log).
- "VALLE SAGRADO VIP" vs "Valle sagrado vip" (ROADMAP Phase 02.1, success criterion #8 context) — motivación para catálogos CRUD en lugar de texto libre. El planner debe asegurar que el form de venta use selects cargados desde los catálogos, no inputs text.
- TC preferencial "según necesidad" — la admin decide cuándo y a qué TC operar. No hay TC histórico ni tabla tipo_cambio. El asiento de TC interno es el único momento donde un valor de TC entra al sistema.
- Los vendedores tienen "perfil" / acuerdo con admin — `comision_reglas` con `vendedor_id` + `tour_id` nullable + precedencia 1-4 captura esto naturalmente. El endpoint `/simular` deja probar reglas antes de guardar — importante para que la admin tenga confianza al cambiar un split.
- Módulo café/Basílica — la separación core/módulo Tours (SC#9) es estructural, no cosmetic. `usuarios`, `contactos`, `cuentas`, `asientos`, `asiento_lineas`, `audit_log`, `modulos` (core) + `tours_catalogo`, `liquidaciones`, `tours_servicios`, `comision_reglas` (módulo). futuro módulo café agrega sus tablas a `apps/cafe/` sin tocar el core, pero puede reusar `contactos`, `cuentas`, `asientos`, `audit_log` (no duplica).
- `next.config.ts` output: `"standalone"` para que el Dockerfile sea más liviano (sirve solo `.next/standalone` + `.next/static` + `public`). Runtime node:22-slim.
